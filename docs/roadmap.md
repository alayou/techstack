# Techstack 路线图

## 目标

Techstack 的长期目标，不只是做仓库分析，而是成为 Code Agent 可直接查询、可被开发者约束、可持续同步更新的技术栈治理层。

它最终要回答的不是“有哪些仓库和包”，而是：

- 这个团队当前真正使用什么技术栈
- 开发者更偏好哪些仓库、包和实现方式
- 哪些候选包或仓库值得推荐，但还没有被正式采纳
- 哪些依赖存在安全风险，不适合继续推荐
- Agent 在具体项目里应该优先参考哪些仓库、哪些包、哪些 skill

## 当前基线

基于当前仓库，已经有这些能力作为后续规划基础：

- 已有公共仓库导入能力
- 已有全局包目录与版本同步能力
- 已有仓库依赖扫描与 AI 技术分析能力
- 已有异步任务机制
- 已有仓库收藏能力，但目前主要停留在“收藏记录”，还没有真正变成技术偏好引擎

当前仍缺失的关键能力包括：

- MCP 服务层
- 包收藏与统一偏好模型
- AI 技术栈仓库归类体系
- Agent 自动发现优秀包和仓库
- 安全数据关联与安全分析
- 面向具体项目的 skill 生成
- 技术栈与偏好的近实时同步更新
- Code Agent 按偏好优先选型的约束机制

## 版本阶段

### v0.2 偏好与分类基础

目标：先把“用户偏好”从零散收藏行为变成系统可用的结构化信号。

计划交付：

- 保留现有仓库收藏能力，并补齐“收藏即偏好”的建模
- 增加包收藏能力
- 增加开发者技术偏好聚合视图
- 建立 AI 技术栈仓库分类字段、标签和基础归档能力
- 为后续安全分析与推荐系统预留数据结构

完成标志：

- 系统可以同时表达“收藏仓库”“收藏包”“偏好来源”
- Agent 或前端可以查询某个用户的偏好仓库与偏好包
- 仓库可以被归类，而不是只靠名称和描述检索

### v0.3 MCP 接入层

目标：把 Techstack 变成 Agent 能直接调用的标准服务，而不是只暴露 HTTP 页面接口。

计划交付：

- 增加 MCP 服务
- 提供查询包、查询仓库、查询技术分析、查询偏好、查询安全信息的 MCP tools
- 补齐 Agent 使用场景需要的鉴权、审计和限流设计
- 明确 MCP 输出的数据结构，避免返回过于松散的自然语言

完成标志：

- IDE 或 Code Agent 可以通过 MCP 查询 Techstack
- 一个 Agent 在写代码前，可以先读取偏好包、偏好仓库、仓库分析和安全信息

### v0.4 推荐与安全治理

目标：让系统不仅能被动记录，还能主动发现候选技术栈并给出安全视角。

计划交付：

- Agent 自动发现优秀包和仓库
- 对候选包和仓库建立可审查推荐结果，而不是直接采纳
- 关联安全数据源，补齐漏洞、风险、受影响版本等安全信息
- 在包、版本、仓库三个层面输出基础安全分析
- 让推荐排序同时参考质量信号、偏好信号和安全信号

完成标志：

- 用户可以查看“推荐但未采纳”的候选包和仓库
- 用户可以查看某个包或仓库的安全信息和基础风险判断

### v0.5 Skill 生成与 Agent 约束执行

目标：把技术栈知识真正接入具体开发行为。

计划交付：

- 基于项目上下文生成可供 Code Agent 使用的 skill
- 把仓库分析、偏好、分类和安全信息打包为 Agent 可消费的上下文
- 建立 Code Agent 的优先选型策略
- 在生成代码时优先选择开发者认可或收藏的包与仓库
- 对不在偏好范围内的候选技术给出显式降级或警告策略

完成标志：

- 给定一个项目，系统可以产出面向该项目的 skill 或上下文包
- Agent 在生成代码时能体现“优先使用偏好包/仓库”的策略

### v0.6 近实时同步

目标：让技术栈知识不只是静态快照，而是持续更新的运行时知识层。

计划交付：

- 依赖变化后的增量同步
- 收藏变化后的偏好快照刷新
- 推荐结果的重算
- 安全数据定时同步与风险状态更新
- 面向 Agent 的缓存更新与失效策略

完成标志：

- 仓库、依赖、收藏和安全数据变化后，相关查询结果可以在较短时间内更新
- Agent 读取到的技术栈信息与开发者当前偏好保持基本一致

## 近期待办边界

在前几个版本里，暂时不把下面这些能力作为主目标：

- 自动替开发者安装或升级依赖
- 自动修改仓库并直接提交上线
- 在没有开发者确认的情况下自动采纳新技术栈
- 做成“万能编程助手”，脱离技术栈治理定位
